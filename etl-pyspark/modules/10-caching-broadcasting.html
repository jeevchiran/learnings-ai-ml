When to Cache

Spark's lazy evaluation means that every time you call an action on a DataFrame, Spark re-executes the entire lineage of transformations from the source data. This is usually fine for one-off computations, but becomes wasteful when the same DataFrame is used in multiple subsequent operations. Caching tells Spark to store the computed result in memory (or disk) so that subsequent actions can skip the recomputation.

Cache when you have reused DataFrames — the same DataFrame feeding into multiple downstream operations. Iterative algorithms like ML training loops are a classic case: each iteration reads the same training data, so caching it avoids redundant I/O and computation. Expensive intermediate results — the output of a heavy join, complex aggregation, or multi-step cleaning pipeline — are also prime candidates.

The rule is simple: if a DataFrame is used more than once after an expensive computation, cache it. If it's used exactly once, caching wastes memory without providing any benefit. The cost of caching is the memory (or disk space) used to store the result, so only cache DataFrames that justify that cost through repeated reuse.

cache() vs persist()

PySpark provides two methods for caching DataFrames. The cache() method is a shorthand for persist(StorageLevel.MEMORY_ONLY) — it stores the entire DataFrame in memory as deserialized Java objects. The persist() method gives you control over the storage level.

Available storage levels:

  • MEMORY_ONLY — fastest access, but the operation fails if the data doesn't fit in available memory. Partitions that don't fit are recomputed on the fly.
  • MEMORY_AND_DISK — the safe default. Partitions that don't fit in memory spill to local disk. Slightly slower for spilled partitions, but never loses data.
  • DISK_ONLY — slowest, stores everything on disk. Useful when you need to preserve memory for other operations but still want to avoid recomputation.
  • MEMORY_ONLY_SER — stores serialized bytes in memory. Uses less memory than MEMORY_ONLY but requires CPU to deserialize on each read.

Unpersisting

When you're done with a cached DataFrame, call unpersist() to free the memory. This is important for long-running applications (Spark Streaming, notebook sessions, iterative jobs) where cached data can accumulate and starve other operations of resources.

Broadcast Variables

When you join a large DataFrame with a small one, Spark's default strategy shuffles data across the network — partitions from both sides get redistributed so that matching keys land on the same node. This is expensive for large datasets. Broadcast joins solve this: the small table is sent (broadcast) to every worker node, so each partition of the large table can do a local lookup without any shuffle.

Use broadcast() from pyspark.sql.functions to hint that a DataFrame is small enough to broadcast. Good candidates include lookup tables, dimension tables, configuration mappings, and any DataFrame under approximately 100MB.

Accumulator Variables

Accumulators let you aggregate values across worker nodes — counting events, summing metrics, or tracking errors during distributed processing. The driver creates an accumulator, workers add to it during transformations, and only the driver can read the final value.

Key rules: workers can only add() to an accumulator (they cannot read it). The driver reads the value after an action completes. Accumulators inside transformations may be updated more than once if tasks are retried, so use them for debugging and monitoring rather than critical logic.

Stretch — Problem 4

For each scenario, choose the appropriate storage level and explain why:

(a) A DataFrame used 10 times that fits comfortably in memory.

MEMORY_ONLY — since it fits in memory, use the fastest option. No need to spill to disk or serialize. Ten reuses means the memory cost is easily justified by the computation savings.

(b) A large DataFrame used 3 times that doesn't fit entirely in memory.

MEMORY_AND_DISK — the partitions that fit in memory get fast access, while the overflow spills to local disk rather than being recomputed. Three reuses justify the disk I/O overhead for spilled partitions.

(c) A DataFrame used twice where memory is scarce and needed for other operations.

DISK_ONLY — preserves memory for the other operations that need it. Two reuses still save time compared to full recomputation, and disk reads are faster than rerunning an expensive pipeline from scratch.
