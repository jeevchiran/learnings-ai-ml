Module 3: Logistic Regression

Bridge from Modules 1-2: Linear regression models continuous outcomes. But many real problems involve categorical decisions: Is this email spam? Does this patient have diabetes? Which digit is written here? Linear regression fails for classification -- it predicts values outside [0,1] and has no principled way to represent uncertainty about discrete outcomes. We need a model that outputs probabilities.

Why Linear Regression Fails for Classification

Consider binary classification where y \in \{0, 1\}. If we naively fit a linear regression \hat{y} = \mathbf{x}^T\boldsymbol{\beta}, the predictions can be any real number -- including values less than 0 or greater than 1. These cannot be interpreted as probabilities.

Linear regression applied to binary data: predictions exceed [0,1] bounds.

The Logistic Model

The solution: pass the linear predictor through a function that maps \mathbb{R} \to (0, 1). The sigmoid (logistic) function:

\sigma(z) = \frac{1}{1 + e^{-z}}

The logistic regression model is:

P(y = 1 \mid \mathbf{x}) = \sigma(\mathbf{x}^T\boldsymbol{\beta}) = \frac{1}{1 + e^{-\mathbf{x}^T\boldsymbol{\beta}}}

The Log-Odds (Logit) Interpretation

Taking the inverse of sigmoid gives the logit function:

\log\frac{p}{1-p} = \mathbf{x}^T\boldsymbol{\beta} = \beta_0 + \beta_1 x_1 + \cdots + \beta_p x_p

The model is linear in log-odds but nonlinear in probability. Each unit increase in x_j changes the log-odds by \beta_j, which multiplies the odds by e^{\beta_j}.

Odds, Log-Odds, and the Logit Function

Probability expresses the chance of an event occurring, bounded between 0 and 1. But this bounded range is inconvenient for a linear model -- we need a quantity that can range over all real numbers.

Odds transform probability into the range (0, \infty):

\text{Odds} = \frac{p}{1 - p}

If p = 0.75, the odds are \frac{0.75}{0.25} = 3, meaning the event is 3 times more likely to happen than not. Odds of 1 correspond to p = 0.5 (a coin flip).

Log-odds (logit) take the natural logarithm of odds, mapping to (-\infty, +\infty):

\text{logit}(p) = \log\frac{p}{1 - p} \in (-\infty, +\infty)

This is the transformation that makes logistic regression "linear" -- the logit of the probability is a linear function of the features.

Why Work in Log-Odds Space?

Three reasons make the logit the natural scale for classification models:

1. It linearizes the problem. Probability is bounded and its relationship with features is typically S-shaped. The logit transformation straightens this curve, allowing us to model it as a simple linear combination \beta_0 + \beta_1 x_1 + \cdots. Without this transformation, a linear predictor could produce impossible probabilities outside [0, 1].

2. Coefficients become multiplicative on odds. In log-odds space, a coefficient \beta_j has a clean interpretation: a one-unit increase in x_j adds \beta_j to the log-odds. Exponentiating gives the odds ratio e^{\beta_j} -- the factor by which the odds are multiplied. This is directly actionable in domains like medicine and business.

3. It arises naturally from the Bernoulli distribution. The logit is the canonical link function for the Bernoulli family in the exponential family framework. It is not an arbitrary choice -- it is the mathematically natural way to parameterize binary outcomes (derived fully in Module 4).

The Symmetry Property

The logit function is symmetric around zero: \text{logit}(p) = -\text{logit}(1-p). This means swapping the labels (calling "success" what was "failure") simply negates all coefficients. The model is inherently symmetric with respect to the two classes.

How Log-Odds Are Used in Practice

Reading model output. Most software reports logistic regression coefficients on the log-odds scale. A coefficient of \beta_1 = 0.7 means: for each unit increase in x_1, the log-odds of the positive class increase by 0.7, or equivalently the odds are multiplied by e^{0.7} \approx 2.01.

Converting between scales. The three representations are interconvertible:

p = \frac{1}{1 + e^{-\text{logit}}} \qquad \text{Odds} = e^{\text{logit}} \qquad \text{logit} = \log(\text{Odds})

Clinical example: diabetes risk. A model predicts diabetes with:

\text{logit}(P(\text{diabetes})) = -6.0 + 0.05 \cdot \text{BMI} + 0.03 \cdot \text{Age} + 1.2 \cdot \text{FamilyHistory}

For a patient with BMI = 30, Age = 50, FamilyHistory = 1:

1

Compute log-odds: -6.0 + 0.05(30) + 0.03(50) + 1.2(1) = -6.0 + 1.5 + 1.5 + 1.2 = -1.8

2

Convert to odds: e^{-1.8} = 0.165 (for every 1 person who gets diabetes, about 6 do not)

3

Convert to probability: p = \frac{1}{1 + e^{1.8}} = \frac{0.165}{1.165} = 0.142 (14.2% risk)

4

Interpret family history coefficient: e^{1.2} = 3.32, so having family history multiplies diabetes odds by 3.3x compared to otherwise identical patient without.

Where you see logits in practice:

  • Scikit-learn's decision_function() returns raw logits (log-odds before sigmoid)
  • Neural network classification layers output logits; softmax/sigmoid is applied separately
  • Medical literature reports odds ratios (e^{\beta}) rather than raw coefficients
  • Betting odds (e.g., "3 to 1 against") are odds in the non-log scale

One-vs-Rest (OvR)

Train K separate binary classifiers. Classifier k learns to distinguish class k from "everything else." At prediction time, run all K classifiers and assign the class with highest confidence:

\hat{y} = \arg\max_k \; \sigma(\mathbf{x}^T\boldsymbol{\beta}_k)

For MNIST digit recognition: 10 binary classifiers, each asking "is this digit k?"

Softmax (Multinomial Logistic Regression)

A single model that jointly estimates probabilities for all K classes:

P(y = k \mid \mathbf{x}) = \frac{e^{\mathbf{x}^T\boldsymbol{\beta}_k}}{\sum_{j=1}^K e^{\mathbf{x}^T\boldsymbol{\beta}_j}}

Probabilities sum to 1 by construction. This is the natural generalization of the sigmoid to multiple classes (with K=2, softmax reduces to standard logistic regression).

Regularization

With many features, logistic regression can overfit -- it finds coefficients that perfectly separate the training data but generalize poorly. Regularization adds a penalty to the loss function:

J_{\text{reg}}(\boldsymbol{\beta}) = J(\boldsymbol{\beta}) + \lambda \cdot \text{penalty}(\boldsymbol{\beta})
L1 (Lasso)

Penalty: \lambda \sum_{j=1}^p |\beta_j|

Effect: drives some coefficients to exactly zero, performing automatic feature selection. Useful when you suspect many features are irrelevant.

L2 (Ridge)

Penalty: \lambda \sum_{j=1}^p \beta_j^2

Effect: shrinks all coefficients toward zero but never exactly to zero. Stabilizes estimates when features are correlated.

Elastic Net

Penalty: \lambda_1 \sum|\beta_j| + \lambda_2 \sum\beta_j^2

Combines L1 and L2. Gets sparsity of L1 with the stability of L2 when features are grouped/correlated.

No Penalty (\lambda = 0)

Unregularized MLE. Works well when n \gg p (many more samples than features). Overfits when p is large relative to n.

Cross-Validation

Never tune hyperparameters on training data -- this leads to overfitting the validation strategy. Use k-fold cross-validation:

\text{CV Score} = \frac{1}{K}\sum_{k=1}^K \text{Score}(\hat{f}_{-k}, \text{fold}_k)

where \hat{f}_{-k} is the model trained on all folds except k.

Evaluation Metrics for Classification

Unlike regression where we have a single continuous error, classification requires multiple metrics because different types of mistakes carry different costs.

The Confusion Matrix

All classification metrics derive from four fundamental counts:

\begin{array}{c|cc} & \text{Predicted Positive} & \text{Predicted Negative} \\ \hline \text{Actual Positive} & \text{TP (True Positive)} & \text{FN (False Negative)} \\ \text{Actual Negative} & \text{FP (False Positive)} & \text{TN (True Negative)} \end{array}

Core Metrics

1

Accuracy -- overall fraction of correct predictions:

\text{Accuracy} = \frac{TP + TN}{TP + TN + FP + FN}
Misleading when classes are imbalanced. A model predicting "no fraud" for every transaction achieves 99.9% accuracy on a dataset with 0.1% fraud rate.
2

Precision -- of all positive predictions, how many are correct:

\text{Precision} = \frac{TP}{TP + FP}
High precision means few false alarms. Critical when the cost of acting on a false positive is high (e.g., spam filter blocking legitimate email).
3

Recall (Sensitivity, True Positive Rate) -- of all actual positives, how many did we catch:

\text{Recall} = \frac{TP}{TP + FN}
High recall means few missed positives. Critical when missing a positive case is dangerous (e.g., cancer screening, fraud detection).
4

F1 Score -- harmonic mean of precision and recall:

F_1 = 2 \cdot \frac{\text{Precision} \cdot \text{Recall}}{\text{Precision} + \text{Recall}} = \frac{2 \cdot TP}{2 \cdot TP + FP + FN}
Balances both concerns. The harmonic mean penalizes extreme imbalance: if either precision or recall is very low, F1 will be low regardless of the other.
5

Specificity (True Negative Rate) -- of all actual negatives, how many did we correctly identify:

\text{Specificity} = \frac{TN}{TN + FP} = 1 - \text{FPR}
Specificity and Recall together define the ROC curve. FPR = 1 - Specificity is the x-axis of the ROC plot.

The Precision-Recall Tradeoff

Precision and recall are inversely related through the threshold. Lowering the threshold (predict positive more readily) increases recall but decreases precision. The optimal balance depends on the cost structure of the problem:

ApplicationPriorityThreshold Strategy
Cancer screeningHigh recall (catch every case)Low threshold (0.2-0.3)
Spam filteringHigh precision (don't lose real email)High threshold (0.7-0.9)
Credit approvalBalance (F1)Threshold at F1 maximum
Fraud detectionHigh recall + acceptable precisionThreshold at target recall with tolerable FPR

Practice Problems: Classification Metrics

Problem 1 (Basic): Computing metrics from a confusion matrix

A disease screening test on 1000 patients produces: TP = 45, FP = 55, FN = 5, TN = 895.

Compute: Accuracy, Precision, Recall, Specificity, F1 Score. Is this a good test? For whom?

Solution

Accuracy = (45+895)/1000 = 94.0%

Precision = 45/(45+55) = 45.0% (many false positives)

Recall = 45/(45+5) = 90.0% (catches most cases)

Specificity = 895/(895+55) = 94.2%

F1 = 2(0.45)(0.90)/(0.45+0.90) = 0.60

Assessment: High accuracy is misleading (only 5% prevalence). The test has excellent recall (misses only 10% of cases) but poor precision (55% of positive results are false alarms). This is acceptable for a screening test where false positives lead to further testing, but unacceptable if a positive result triggers expensive treatment directly.

Problem 2 (Intermediate): Choosing the right metric

You are building three models for different applications. For each, state which metric you would optimize and explain why:

  1. A system that automatically blocks suspicious login attempts
  2. A model that flags manufacturing defects on an assembly line (defects are rare, ~0.5%)
  3. A recommendation system that suggests articles to readers
Solution
  1. Precision-focused (or balanced F1). Blocking a legitimate user (FP) is highly disruptive -- they may leave the platform. Missing one suspicious login (FN) is concerning but less immediately damaging if you have other detection layers. Optimize for precision at a minimum recall threshold (e.g., precision > 0.95 subject to recall > 0.70).
  2. Recall-focused. A defective product reaching the customer (FN) causes safety issues and costly recalls. A false alarm (FP) only means an extra inspection. Optimize recall subject to a tolerable false positive rate. Use F2 score (weights recall 2x higher) or set a target recall of 99%+.
  3. Precision@K or NDCG. Users see the top-K recommendations. Showing an irrelevant article (FP) wastes user attention and reduces trust. Missing a good article (FN) is invisible -- there are always more articles. Precision in the top-K positions matters most.
Problem 3 (Advanced): The F-beta score and cost-sensitive thresholds

A fraud detection model outputs probabilities for 10,000 transactions. The cost matrix is: missed fraud (FN) costs $500 average loss; false alarm (FP) costs $10 in investigation time. The model's precision-recall at threshold 0.5 is: P=0.80, R=0.60.

Questions:

  1. What is the appropriate \beta for the F_\beta score given this cost ratio? (The F_\beta score is: F_\beta = (1+\beta^2)\frac{P \cdot R}{\beta^2 P + R})
  2. If the fraud rate is 2% (200 frauds in 10,000), compute the total expected cost at threshold 0.5.
  3. At threshold 0.3, precision drops to 0.40 and recall rises to 0.90. Is this threshold better in terms of total cost?
Solution
  1. \beta^2 = \frac{\text{cost of FN}}{\text{cost of FP}} = \frac{500}{10} = 50, so \beta = \sqrt{50} \approx 7.07. This heavily weights recall, reflecting that missing fraud is 50x costlier than investigating a false alarm.
  2. At threshold 0.5: Recall=0.60 means TP=120, FN=80. Precision=0.80 means TP=120, FP=30. Total cost = 80 * $500 + 30 * $10 = $40,300.
  3. At threshold 0.3: Recall=0.90 means TP=180, FN=20. Precision=0.40 means FP=TP*(1-P)/P = 180*0.6/0.4 = 270. Total cost = 20 * $500 + 270 * $10 = $10,000 + $2,700 = $12,700. The lower threshold reduces total cost by 68% despite much lower precision. The cost structure makes recall far more valuable.

Residual Analysis for Logistic Regression

Unlike linear regression where raw residuals (y - \hat{y}) are continuous and interpretable, logistic regression residuals require special treatment because the response is binary.

Types of Residuals

Deviance Residuals

d_i = \text{sign}(y_i - \hat{p}_i)\sqrt{-2[y_i\log\hat{p}_i + (1-y_i)\log(1-\hat{p}_i)]}

The default in most software. Analogous to standardized residuals in linear regression. Should be roughly normally distributed if the model is correct.

Pearson Residuals

r_i = \frac{y_i - \hat{p}_i}{\sqrt{\hat{p}_i(1-\hat{p}_i)}}

Standardized by the binomial variance. Useful for detecting outliers (values beyond +/- 2 are suspicious).

Response Residuals

e_i = y_i - \hat{p}_i

Simple but hard to interpret visually since y is 0 or 1. Creates two horizontal bands in scatter plots. Used mainly for binned residual plots.

Binned Residuals

Group observations by predicted probability, compute average residual per bin. If model is correct, approximately 95% of bins should have average residuals within \pm 2\text{SE}.

Solution

(a) Observations A and B have small deviance residuals (|d| < 1), meaning the model's predicted probability aligns well with the outcome.

(b) Observations C, D, and E are surprising. C defaulted but was given only 15% probability. D did not default despite 85% predicted risk. E defaulted despite the model being nearly certain (2%) it would not.

(c) Observation E has a deviance residual of 2.80 (> 2 standard deviations). This could indicate: (i) a genuinely unusual case (rare event), (ii) an important predictor is missing from the model that would explain this default, or (iii) data error (wrong label). Investigate by examining E's feature values -- is it unlike any training example? Are there features not in the model that differ?

Problem 3 (Advanced): Hosmer-Lemeshow test and model comparison

Two models for disease prediction are evaluated on 500 test patients. The Hosmer-Lemeshow goodness-of-fit test (10 bins) gives:

  • Model A: \chi^2 = 6.2, df = 8, p = 0.63
  • Model B: \chi^2 = 18.7, df = 8, p = 0.017

Model B has higher AUC (0.88 vs 0.82). Both have similar log-loss.

Questions:

  1. Interpret the Hosmer-Lemeshow results for each model.
  2. Model B has better discrimination (AUC) but worse calibration. In what scenario would you still prefer Model B?
  3. Can you have good discrimination with bad calibration? Explain with an example.
Solution
  1. Model A: p = 0.63 means we fail to reject the null (model is well-calibrated). The predicted probabilities agree with observed frequencies. Model B: p = 0.017 means we reject the null -- the model is statistically significantly miscalibrated. Predicted probabilities do not match observed rates.
  2. Prefer Model B when the task is ranking/discrimination only -- e.g., "which patients are at highest risk?" for resource allocation. AUC measures ranking ability, not probability accuracy. If you need to compare the predicted probability to a fixed threshold or communicate risk levels to patients, Model A is better because its probabilities are trustworthy. Model B can also be preferred if you recalibrate it post-hoc (Platt scaling), preserving its superior discrimination while fixing calibration.
  3. Yes. Consider a model that predicts 0.99 for all positives and 0.98 for all negatives. It perfectly separates the classes (AUC = 1.0) but is terribly calibrated -- a prediction of 0.98 does not correspond to 98% probability of the positive class; it corresponds to 0%. Discrimination depends only on the ranking of scores; calibration depends on their absolute values.

Where Logistic Regression Reaches Its Limits

Linear Decision Boundaries

The model can only separate classes with a hyperplane. XOR-like patterns or spiral-shaped classes require nonlinear methods (kernel tricks, neural networks).

Feature Engineering Dependence

Performance depends heavily on the features you provide. Unlike deep learning, logistic regression cannot learn feature representations from raw data.
