Module 8: Multivariate Analysis

What, Why, and How

WHAT: Examining how 3+ variables interact simultaneously

Multivariate analysis goes beyond pairs of columns to examine how three or more variables interact at once. The core tools are:

  • Pivot table — a cross-tabulation that places one variable on rows, another on columns, and summarises a third variable (e.g., mean, sum, count) in each cell. The pandas function is df.pivot_table().
  • Multi-level groupby — df.groupby([col1, col2]).agg() groups by two (or more) keys simultaneously and returns a Series with a MultiIndex, giving you every combination of group values.
  • Color-encoded scatter plot — a standard scatter plot (x vs y) where point color encodes a third categorical variable. Three dimensions of information in one chart: position-x, position-y, and hue.
  • Faceted plots — seaborn's FacetGrid repeats the same chart once per category value, creating a grid of small multiples that let you compare a relationship across subgroups.

WHY: Real phenomena are multivariate — two variables are rarely enough

Bivariate analysis answers "does A relate to B?" But most real-world questions are richer:

  • Does fare_amount depend on trip_distance and RatecodeID and time of day — all at once?
  • Do higher passenger_count trips earn systematically different tips across different rate codes?
  • Is the bivariate correlation between distance and fare actually a masking of two completely different fare regimes (metered vs flat-rate) that behave differently?

In Module 7 we saw that fare_amount and trip_distance correlate well for metered rides — but airport flat-rate trips (RatecodeID=2) break that pattern. The only way to see this clearly is to introduce RatecodeID as a third variable.

HOW: pivot_table, multi-level groupby, and color-encoded scatter

1
Pivot table via pandas
df.pivot_table(values='fare_amount', index='RatecodeID', columns='passenger_count', aggfunc='mean') — rows = unique RatecodeID values, columns = unique passenger_count values, each cell = mean fare for that combination. aggfunc can also be 'sum', 'count', np.median, or a custom lambda.
2
Multi-level groupby
df.groupby(['RatecodeID', 'passenger_count'])['fare_amount'].mean().unstack() — produces the same result as the pivot table when aggfunc is mean. The .unstack() call pivots the inner index level into columns, turning the flat MultiIndex Series into a 2-D DataFrame.
3
Color-encoded scatter (seaborn)
sns.scatterplot(data=df, x='trip_distance', y='fare_amount', hue='RatecodeID', palette='tab10', alpha=0.7) — each unique RatecodeID gets a distinct color from the palette. Points belonging to different rate codes visually separate into clusters, revealing the third-variable effect instantly.

NaN cells in pivot tables are informative, not errors. A NaN in a pivot table cell means no row matched that row×column combination — e.g., RatecodeID=2 with passenger_count=3 never occurred in the sample. Always check whether NaN means "missing data" or "that combination does not exist."

Practice


  
  

    

      
Problem 1: Pivot table aggregation function

      Easy
    

    

      
What pandas function creates a cross-tabulation of means, and what parameter controls the aggregation?

      
      

        
df.pivot_table(values=..., index=..., columns=..., aggfunc='mean')

        
The aggfunc parameter controls how values in each cell are combined. Common options:

        
    
          
  • 'mean' — average of all matching rows
    • 
          
  • 'sum' — total of all matching rows
    • 
          
  • 'count' — number of matching rows (useful for seeing how many observations back each cell)
    • 
          
  • np.median, np.std, or a custom lambda for other statistics
    • 
        

        
The equivalent using groupby: df.groupby([index_col, col_col])[value_col].mean().unstack().

      

    

  


  
  

    

      
Problem 2: Interpreting the color-encoded scatter

      Medium
    

    

      
In the color-encoded scatter (trip_distance vs fare_amount, colored by RatecodeID), why do RatecodeID=2 points cluster differently from RatecodeID=1 points?

      
      

        
RatecodeID=2 trips are flat-rate airport fares. NYC taxi rules fix the fare to Manhattan from JFK at $52 and to Newark (EWR) at $45 — regardless of the exact mileage driven on that specific trip. The fare does not grow with distance the way metered (RatecodeID=1) fares do.

        
This means:

        
    
          
  • RatecodeID=2 points sit at constant y-values ($45 or $52) even though their x-values (trip_distance) vary.
    • 
          
  • RatecodeID=1 points follow a rising diagonal because each additional mile adds a metered increment to the fare.
    • 
        

        
The "third variable" RatecodeID is a confounder: without it, the bivariate scatter shows a messy cluster in the upper-right that looks like a noisy extension of the metered pattern, when in reality it is an entirely different pricing rule.

      

    

  


  
  

    

      
Problem 3: Designing a multivariate hypothesis test

      Hard
    

    

      
You hypothesise that high passenger_count trips have lower average tip_amount per person. Design a multivariate analysis using our dataset to confirm or refute this hypothesis.

      
      

        
A systematic approach:

        
    
          
  1. Filter to card payments only — cash trips have tip_amount=0 by definition (not electronically captured), which would confound the comparison. Use df[df.payment_type == 'card'].
    2. 
          
  2. Compute per-person tip — add a derived column: df['tip_per_person'] = df['tip_amount'] / df['passenger_count']. Watch for NaN when passenger_count is null.
    3. 
          
  3. Groupby passenger_count — df.groupby('passenger_count')['tip_per_person'].agg(['mean', 'count']). The count column tells you how many trips back each mean, so you know whether small groups are unreliable.
    4. 
          
  4. Build a pivot table for a multivariate view — df.pivot_table(values='tip_per_person', index='passenger_count', columns='RatecodeID', aggfunc='mean'). This reveals whether the pattern holds within each rate code, or only appears due to airport trips dominating certain passenger_count values.
    5. 
          
  5. Interpret cautiously — with only 30 rows, most passenger_count groups have very few observations. State the finding as directional, not conclusive, and note the sample size next to each mean.
    6. 
        

        
In the dataset, 3-person and 4-person card trips tend to be group leisure travel (Brooklyn/Queens to Manhattan) where split-tip norms may differ from solo commutes. The pattern is real directionally but requires a larger sample to confirm statistically.

Module 8 — Check Your Understanding


    
    

      
1. A pivot table with aggfunc='mean' shows:

      

        

          A
          Total sum per group
        

        

          B
          Count of rows per group
        

        

          C
          Average value per row×column combination
        

        

          D
          Median per group
        

      

      
Correct. aggfunc='mean' takes all rows that match the (index_val, column_val) combination and computes their arithmetic mean. For sum use 'sum', for count use 'count', for median use np.median. The aggregation function is completely configurable — you can even pass a dict to apply different functions to different value columns.

      
Not quite. aggfunc='mean' computes the average value, not the total or the count. For total use 'sum'; for the number of rows use 'count'; for the median use np.median. The key idea is that each cell summarises potentially many rows using the chosen aggregation function.

    


    
    

      
2. In a color-encoded scatter plot, a third variable is encoded as:

      

        

          A
          Point size only
        

        

          B
          X-axis position
        

        

          C
          Point color (hue)
        

        

          D
          Line style
        

      

      
Correct. Using color (hue) to encode a third variable is the standard "color-encoded scatter" technique. Seaborn uses hue= for this. A fourth variable can be added via size= (point size) and a fifth via style= (marker shape), though more than three visual channels quickly becomes hard to interpret.

      
Not quite. X-axis encodes the first variable, Y-axis the second — those are the two spatial channels. The third variable is mapped to point color (called "hue" in seaborn). Point size (size=) and marker style (style=) can encode a fourth and fifth variable respectively, but color is the most perceptually salient channel for a third discrete variable.

    


    
    

      
3. Why is multivariate analysis harder to interpret than bivariate?

      

        

          A
          It requires more data
        

        

          B
          Computers cannot handle it
        

        

          C
          Interactions between variables can mask or amplify pairwise relationships
        

        

          D
          It is not harder — just use more charts
        

      

      
Correct. This is called Simpson's Paradox and related interaction effects. A trend that appears in the overall bivariate view can reverse, disappear, or strengthen when you condition on a third variable. In our taxi data, the apparent correlation between distance and fare hides two completely different pricing rules. Adding RatecodeID as a third variable reveals the truth, but it also introduces the need to reason about conditional relationships — which is genuinely more complex.

      
Not quite. More data is helpful but not the reason for complexity. Computers handle multivariate data fine. And adding more charts is not sufficient — the key challenge is that interactions between variables can reverse or mask pairwise relationships. A variable that predicts Y strongly in one group may have no effect in another. This conditional reasoning is what makes multivariate analysis harder than simply examining pairs.

    


    
    

      
4. df.groupby(['RatecodeID','passenger_count'])['fare_amount'].mean() returns:

      

        

          A
          A scalar value
        

        

          B
          A flat list
        

        

          C
          A Series with a MultiIndex
        

        

          D
          A pivot table
        

      

      
Correct. When groupby receives a list of columns, the resulting index has one level per grouping column — this is a MultiIndex. The result is a pandas Series where each entry is the mean fare for a unique (RatecodeID, passenger_count) pair. To convert this into a 2-D table (equivalent to pivot_table output), chain .unstack(), which moves the innermost index level (passenger_count) into columns.

      
Not quite. .mean() on a multi-column groupby returns a Series with a MultiIndex — one index level per grouping column. It is not a scalar (that would require a single group), not a list (wrong data structure), and not already a pivot table (though .unstack() converts it to one). The MultiIndex lets you do .loc[(ratecode, passenger_count)] to retrieve individual cells.
