Module 5: Datetime Parsing & Feature Engineering

What, Why, and How

WHAT: Parsing Datetimes into Structured Features

A raw datetime string like "2023-03-06 08:15:00" carries rich information — the hour of day, the day of week, the elapsed duration between events — but it sits in the column as an opaque blob of text. Datetime feature engineering is the act of parsing that string into a proper datetime type and then extracting the numeric signals hidden inside.

The three most valuable signals from taxi trip datetimes are:

  • Hour of day (0–23) — captures the within-day demand cycle
  • Day of week (0=Monday, 6=Sunday in pandas convention) — captures weekend vs. weekday patterns
  • Trip duration (minutes) — derived by subtracting pickup from dropoff, exposing how long each trip actually ran

WHY: Raw Strings Are Useless for Models and Aggregations

You cannot compute mean("2023-03-06 08:15:00") — it is a string. You cannot group by "rush hour" unless you first extract the hour. Models trained on the raw string would either fail or — worse — silently treat it as a categorical label, missing all temporal structure.

Once you extract hour and day_of_week, patterns emerge immediately:

  • Rush hour demand: rides spike at 8–9 AM and 5–7 PM
  • Weekend shift: Saturday/Sunday trips tend to be longer and later at night
  • Duration outliers: row 24 shows a 45-minute trip with a suspicious 120-mile distance — computing duration flags it immediately

In ML pipelines, raw datetime strings must be converted before any feature can be computed. Extracting these features is not optional — it is the gateway to the entire time dimension of the data.

HOW: pd.to_datetime() and the .dt Accessor

1
Parse the column
Convert the string column to datetime64 dtype. Pandas auto-detects common formats; pass format= for speed on large data.
2
Extract scalar features via .dt
The .dt accessor unlocks sub-components: .dt.hour, .dt.dayofweek, .dt.month, .dt.is_weekend, and dozens more — each returns a full column of integers.
3
Compute duration as a timedelta
Subtracting two datetime columns gives a timedelta64 column. Chain .dt.total_seconds() / 60 to get minutes as a float.
4
Group and aggregate
Once features are numeric, df.groupby('hour').size() immediately gives rides-per-hour. Temporal patterns become visible in one line.

Code Walkthrough

Python — parse datetime strings and extract features 
# Step 1: parse string columns → datetime64
df['tpep_pickup_datetime']  = pd.to_datetime(df['tpep_pickup_datetime'])
df['tpep_dropoff_datetime'] = pd.to_datetime(df['tpep_dropoff_datetime'])

# Step 2: extract scalar features via .dt accessor
df['hour']        = df['tpep_pickup_datetime'].dt.hour        # 0–23
df['day_of_week'] = df['tpep_pickup_datetime'].dt.dayofweek   # 0=Monday

# Step 3: compute trip duration in minutes
df['duration_min'] = (
    df['tpep_dropoff_datetime'] - df['tpep_pickup_datetime']
).dt.total_seconds() / 60

print(df[['tpep_pickup_datetime', 'hour', 'day_of_week', 'duration_min']].head(4))

Duration formula — the arithmetic is simple but the types matter.
\text{duration\_min} = \frac{t_{\text{dropoff}} - t_{\text{pickup}}}{60}
Subtracting two datetime64 columns yields a timedelta64. Calling .dt.total_seconds() converts the timedelta to a float in seconds; dividing by 60 gives minutes. Without the pd.to_datetime() step first, the subtraction would fail — you cannot subtract strings.

Practice

Problem 1: Converting a String Column

Easy

What pandas method converts a string column to datetime dtype so that the .dt accessor becomes available?

pd.to_datetime(df['col'])

This parses the string column and returns a new Series with datetime64[ns] dtype. You then assign it back: df['col'] = pd.to_datetime(df['col']). Once the dtype is datetime, df['col'].dt.hour, df['col'].dt.dayofweek, and all other .dt sub-accessors become available. Attempting .dt.hour on a string column raises an AttributeError.

Problem 2: Computing Duration

Medium

If tpep_pickup_datetime is 2023-01-15 08:32:00 and tpep_dropoff_datetime is 2023-01-15 09:05:00, what is the trip duration in minutes?

33 minutes

09:05 − 08:32 = 33 minutes. In pandas: (df['tpep_dropoff_datetime'] - df['tpep_pickup_datetime']).dt.total_seconds() / 60 returns 33.0. The subtraction of two datetime64 columns yields a timedelta64; .dt.total_seconds() converts to a float of seconds (1980.0), and dividing by 60 gives minutes.

Problem 3: Why Extract Temporal Features?

Hard

Why is dt.dayofweek more useful for ML than the raw date string? Name two temporal features you would derive for a demand forecasting model.

dt.dayofweek returns a number (0–6) that encodes recurrent weekly structure. The raw date string "2023-03-06" is a unique identifier — a model trained on it learns nothing about Mondays in general, only about that specific Monday. The integer 0 (Monday) appears every week, so the model can generalise across dates with the same weekday structure.

Two strong features for demand forecasting:

  • hour — captures the intra-day demand curve (morning commute, lunch lull, evening peak). Optionally encode cyclically: sin(2π·hour/24) and cos(2π·hour/24) so hour 23 is numerically close to hour 0.
  • is_weekend — a boolean flag (day_of_week >= 5) collapses the 7-day week into two demand regimes (workday vs. leisure day) that behave very differently in most transportation datasets.

Quiz

Module 5 — Check Your Understanding

1. What does pd.to_datetime() return?

A A list of strings
B A Series with datetime64 dtype
C A Unix timestamp integer
D A Python datetime object
Correct. pd.to_datetime() returns a pandas Series (or Index) with datetime64[ns] dtype. This unlocks the .dt accessor and enables timedelta arithmetic between columns. Passing a single string returns a single Timestamp, but on a column it returns a full Series.
Not quite. pd.to_datetime(col) returns a pandas Series with datetime64[ns] dtype — not strings, not raw Unix integers, and not plain Python datetime objects (though each element is representable as one). The dtype change is what enables .dt.hour and timedelta subtraction.

2. In pandas, dt.dayofweek returns 0 for:

A Sunday
B Monday
C Saturday
D The first day of the dataset
Correct. Pandas follows the ISO 8601 convention: 0 = Monday, 1 = Tuesday, …, 6 = Sunday. This differs from Python's datetime.weekday() (same convention) but differs from JavaScript's Date.getDay() which returns 0 for Sunday. Watch for this discrepancy when bridging Python and JS code.
Not quite. Pandas uses ISO convention: 0 = Monday, 6 = Sunday. JavaScript's Date.getDay() does return 0 for Sunday — a common source of off-by-one bugs when converting between the two. In pandas, is_weekend is therefore dayofweek >= 5.

3. Why extract hour and day_of_week as separate features?

A To reduce file size
B To convert strings to numbers
C To expose cyclical temporal patterns that models can learn
D Because pandas requires it for groupby
Correct. The raw date string "2023-03-06" is unique — a model learns nothing reusable from it. But hour=8 appears every day: the model discovers that 8 AM is consistently high-demand. Cyclical features like hour and day_of_week encode recurrent patterns that generalise across dates — that is their fundamental value.
Not quite. The core reason is pattern generalisation: a raw date string is a unique label (not reusable), but hour=8 appears every single day — the model learns that 8 AM is high-demand in general. Extracting these features converts one-time identifiers into recurring signals. String-to-number conversion is a side effect, not the reason.

4. Trip duration is computed as:

A df['hour'].diff()
B df['dropoff'] - df['pickup'] (gives a string)
C (df['dropoff'] - df['pickup']).dt.total_seconds() / 60
D pd.duration(df['dropoff'], df['pickup'])
Correct. Subtracting two datetime64 columns gives a timedelta64 column. The .dt.total_seconds() accessor converts that to float seconds, and dividing by 60 gives minutes. The key prerequisite is that both columns must already be datetime64 — if they are still strings, the subtraction will raise a TypeError.
Not quite. df['hour'].diff() computes differences between consecutive hour values (not duration). Subtracting string columns gives a TypeError, not a string result — you must parse with pd.to_datetime() first. There is no pd.duration() function. The correct chain is: parse both columns → subtract → .dt.total_seconds() / 60.
